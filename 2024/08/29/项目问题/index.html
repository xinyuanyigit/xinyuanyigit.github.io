扫码登录的设计与实现

用户名密码登录：

在登录页面输入用户名和密码，服务端验证用户名和密码，验证失败返回登录页重新登录，验证成功生成token，存储token到redis中，返回token给前端。之后每次请求都会携带该token，验证该token，验证通过，执行后续接口，返回执行结果。验证失败，返回登录页重新登录。

二维码扫码登录：

二维码是PC端、手机端和服务端之间的桥梁。

流程：

  1. 登录页发送请求到服务端，服务端会生成一个唯一ID，并把这个唯一ID设置到redis中，设置它的过期时间。服务端返回这个唯一ID到PC端，PC端会根据这个唯一ID生成一个二维码。在这个步骤里，在redis中存储的是一个以唯一ID为键的值，它有三种状态，如果这个以唯一ID为键的值为null，表示为未扫码，有值的话表示为扫描成功，如果不存在这个key，表示为已失效。
  2. 手机端扫描二维码，解析出唯一ID，然后把appId、手机端token以及解析出的唯一ID传递到服务端，服务端首先会验证手机端的token是否有效，如果无效则验证失败，有效的话会验证唯一ID，判断是否存在唯一ID，如果不存在，表示二维码已经失效了，如果存在且状态为未扫码，则生成一个PC端的token存储在redis中的这个唯一ID为键的值，其他情况都是验证失败。
  3. PC端会以轮询的方式发送异步请求向服务端查询这个唯一ID的状态，如果redis中的唯一ID为键的值为null，表示未扫码，如果这个唯一ID不存在，表示二维码已失效，如果找到这个唯一ID对应的值，表示扫码成功，同时返回这个PC端的token，提示登录成功。

使用redis怎么统计网站的UV？

UV：指一定时间内，访问网站的不同用户的数量。

方法一：使用incr命令。

incr命令是一个简单的key-value格式的计数器，用户每次访问网站，都会根据用户的唯一标识去执行incr命令，最后通过get命令来获取访问数量。但这个方法不能去重，也就是说，同一个用户访问网站两次，incr命令就会使访问量加2，不适用于UV。

方法二：使用set数据类型来设计。

set集合拥有唯一性，可以用于统计去重。通过sadd命令来进行统计访问量。通过scard命令来获取访问量。但这个方法也有一个问题，那就是需要大量内存。

方法三：使用BitMaps类型来设计。

BitMaps是redis中的特殊的数据类型。它用一个bit来表示某个值的状态，只存储0或1，所以它可以节省极大的内存。我们可以通过setbit命令来进行统计访问量，通过bitcount来获取访问量。但如果统计的对象很多的话，还是需要很多内存。

方法四：使用HyperLogLog类型来设计。

HyperLogLog是redis中的特殊的数据类型，它使用极小的内存就可以统计大量的元素。HyperLogLog是通过pfadd命令来添加元素进行统计，使用pfcount命令类进行获取统计量。

为什么使用策略模式来封装不同的评分算法？有什么好处？怎么实现？

策略模式是一种设计模式，它定义了一系列算法，并将每一个算法封装起来，使它们可以相互替换。策略模式的主要目的是为了将算法的使用和实现分离，从而提高系统的灵活性和可维护性。 在策略模式中，通常会定义一个接口或抽象类，表示一组算法，然后实现这个接口或抽象类的不同算法。这些算法被称为策略，它们可以被动态地选择和切换。客户端可以通过选择不同的策略来改变算法的行为，而不需要修改代码。 策略模式的优点是可以减少代码的重复性和复杂性，提高代码的复用性和可维护性。同时，它还可以让系统更加灵活，更容易适应变化。

定义一些独立的类封装不同算法，这里的每一种类中的算法就是一种策略，为了保证这些策略具有一致性，提供一个抽象的策略类来做算法的声明。

策略模式的优点：

策略类之间可以自由切换：由于策略类都实现同一个接口，所以使它们之间可以自由切换。

易于扩展：增加一个新的策略只需要添加一个具体的策略类即可，基本不需要改变原有的代码，符合“开闭原则“。

策略模式使用场景：

一个系统需要动态地在几种算法中选择一种时，可将每个算法封装到策略类中。

策略模式最大的作用在于分离使用算法的逻辑和算法自身实现的逻辑，这样就意味着当我们想要优化算法自身的实现逻辑时就变得非常便捷，一方面可以采用最新的算法实现逻辑，另一方面可以直接弃用旧算法而采用新算法。使用策略模式能够很方便地进行替换。

一个类定义了多种行为，并且这些行为在这个类的操作中以多个条件语句的形式出现，可将每个条件分支移入它们各自的策略类中以代替这些条件语句。

在实际开发中，有许多算法可以实现某一功能，如查找、排序等，通过 if-else 等条件判断语句来进行选择非常方便。但是这就会带来一个问题：当在这个算法类中封装了大量查找算法时，该类的代码就会变得非常复杂，维护也会突然就变得非常困难。虽然策略模式看上去比较笨重，但实际上在每一次新增策略时都通过新增类来进行隔离，短期虽然不如直接写 if-else 来得效率高，但长期来看，维护单一的简单类耗费的时间其实远远低于维护一个超大的复杂类。

系统要求使用算法的客户不应该知道其操作的数据时，可使用策略模式来隐藏与算法相关的数据结构。

如果我们不希望客户知道复杂的、与算法相关的数据结构,在具体策略类中封装算法与相关数据结构,可以提高算法的保密性与安全性。

怎么实现AI生成题目的功能？

  1. 编写Prompt告诉AI模型需要什么样的内容，可以使用自然语言进行格式的描述。
  2. 进行系统预设、少样本学习、任务拆解等技巧优化Prompt，使AI能更好的理解和生成题目。系统预设是指预先设定一些规则或模板；少样本学习是指利用少量已有的题目训练AI模型；任务拆解则是指将复杂的任务分解为多个简单的子任务，逐个解决。
  3. 通过AI平台提供的界面，测试和调整Prompt，直到得到满意的结果。
  4. 为了提高响应速度，使用流式化和SSE技术进行优化。
  5. 流式API允许实时接收AI生成的内容，不用等待整个内容生成完毕再一次返回。
  6. 通过Server-Sent Events (SSE) 技术，让服务器向客户端实时推送数据。用SSE将生成的单道题目实时推送给前端，提高用户体验。

如何流式调用AI并实时获取AI返回的内容？

智谱AI的SDK提供了流式调用的能力，我自主封装了一个AI Manager简化了流式调用的传参，返回一个响应式对象。利用响应式编程框架RXJava来处理AI实时返回的数据，将AI返回的字符串拼接成一个一个题目，通过SSE实时推送给前端。

如何实现一个一个流式生成题目？

利用RxJava链式调用并处理AI实时返回的异步数据。通过map，filter处理字符串，通过flatmap把字符串映射为单个字符，再通过括号平衡算法准确拼接出单道题目。

括号平衡算法：题目是json格式的，左括号数一定等于右括号数。遍历当前字符串，用一个计数器来计数括号数，出现左括号+1，出现右括号-1，直到计数器为0时，表示一道题目，之后通过SSE技术推送题目给前端。

什么是SSE技术？他有什么优点和不足？适用于哪些场景？

SSE技术是一种用于从服务器到客户端的单向、实时数据传输技术，它是基于HTTP协议实现的。

特点：

  • 单向通信：SSE只支持服务器向客户端的单向通行，客户端不能向服务器发送数据。
  • 文本格式：SSE使用纯文本格式传输数据，使用HTTP响应的text/evebt-streamMIME类型。
  • 保持连接：SSE通过保持一个持久的HTTP连接，实现服务器向客户端推送更新。不需要客户端频繁轮询。
  • 自动重连：如果连接中断，浏览器会自动重新连接，保证数据流的连续性。

优点：

  • 简单易用
  • 实时性
  • 标准化
  • 无需握手机制

不足：

  • 单向通信：
  • 连接维持：需要保持持续连接，可能造成服务器资源占用。

使用场景：

  • 实时更新：股票价格，体育比赛比分、新闻更新等需要实时推送的应用。
  • 日志监控：实时监控服务器日志或应用状态。
  • AI对话：常见的AI模型页面对话。

在项目中如何利用缓存？怎么设计的？

在AI评分中存在两个问题：

  • AI调用需要费用，如果用户对同样的应用做出同样的回答，理论会得到一样的回答，不需要每次都询问AI。存在资源浪费的问题。
  • AI评分的响应时间较长，效率有待提高。

考虑到减少响应时间和数据复用，选择用缓存技术来优化AI评分功能。选择成本较低的本地缓存caffeine来实现。

设计缓存key：

用户对同样应用做出一样的回答会得到同样的结果，可以将应用id和用户的答案列表作为key，因为答案列表可能会很长，所以用hash算法来压缩答案列表，节省空间。

value：

缓存AI回答的结果。为了节省空间，存储二进制数据。

缓存过期时间：

必须设置过期时间，为了节省空间。设置过期时间为1天。

业务流程：

  1. 在用户用AI回答前，先hash处理用户答案选择，得到摘要，拼接缓存key。
  2. 通过这个key查询缓存，若命中则直接返回命中结果
  3. 若未命中，则请求AI回答。
  4. 正确解析AI的返回结果，将其放到缓存里。

什么是幂等设计，怎么实现的？

使用相同参数多次调用同一接口，产生的结果或影响和单词调用结果是一样的。

使用数据库唯一索引来保证幂等操作：

流程：

  1. 用户进入答题页面，请求后端返回一个全局唯一id
  2. 用户提交回答时，不仅要提交答案列表，要包括全局唯一id
  3. 后端将这个全局唯一id保存到数据库中的某个唯一索引字段上，利用数据库的唯一索引实现幂等。

为防止用户多次提交重复答案，基于雪花算法为每次答题分配唯一id，并通过数据库主键实现 幂等设计，避免了重复的脏数据。

核心是通过时间戳(亳秒)保证递增，通过机器id 、服务id 和递增序号(同一毫秒内递增)，保证唯一性。

什么是线程池隔离?你在项目中为什么要使用线程池隔离，有什么好处?

线程池隔离是一种通过为不同的任务类型或者不同的资源来分配不同的线程池。这种机制可以有效的控制不同任务之间的影响，提高系统的稳定性和可靠性。

如果所有的业务都使用同一个线程池，最大的问题就是不同任务之间相互影响。所以在一些敏感的场景中，需要隔离线程池，它的好处有：

  1. 故障隔离，缩小事故范围
  2. 资源隔离，防止业务之间抢占资源。同时支持更精细化的管理资源，比如不重要的场景给小一点的线程池，核心场景配置大的线程池。
  3. 性能优化，一些任务的场景是CPU密集型，一些是I/O密集型，不同任务类型需要配置不同的线程池。

在本项目中，因为服务器资源有限，对于AI批量生成题目功能，用户分为会员和普通用户，会员会分配核心线程数更大的线程池，普通用户分配核心线程数更小的线程池。

幂等操作怎么实现？

客户端实现幂等操作：

①重复提交表单 ②像AI智能答题平台调用AI模块进行答题时要保证接口的幂等

实现流程：

  • 前端进入表单提交页面时，向服务端发送获取唯一 token 的请求。
  • 服务端生成唯一的 token，并将其存储到 Redis 中，同时设置过期时间。返回这个 token 给前端。
  • 当页面提交时，前端携带这个 token 发送请求。
  • 服务端接收请求后，开始执行 Redis 事务：
    • 使用 MULTI 开始事务。
    • 尝试获取分布式锁（例如使用 SETNX 操作），如果获取成功，继续执行下一步；否则，说明已经有其他线程持有锁，抛出重复执行的异常。
    • 根据 token 获取 Redis 中的值。如果能成功获取到，说明这是首次提交；否则，说明已经提交过一次，抛出重复执行的异常。
    • 删除 Redis 中的 token。
    • 执行 EXEC 结束事务。
  • 如果事务执行成功，继续执行后续逻辑；如果事务执行失败，抛出重复执行的异常。

服务端实现幂等操作：

①消息的重复消费 ②扣减库存 ③短信的重复发送

实现流程：

  1. 进行库存扣减时，传递的参数是订单id和sku编号，在扣减库存逻辑之前，先获取分布式锁来确保同一时刻只有一个线程能够执行库存扣减操作。

  2. 判断订单id和sku在库存的扣减记录中是否存在，如果存在，则说明已经执行过库存扣减，直接返回成功。

  3. 如果不存在，继续执行库存扣减逻辑：

    • a. 根据订单ID和SKU查询库存表，获取对应商品的库存数量。

    • b. 如果库存数量不足，抛出异常或返回错误信息。

    • c. 如果库存数量足够，执行库存扣减操作。这里可以采用原子性操作，如 MySQL 的 UPDATE 语句配合 WHERE 子句，或者使用 Redis 的 INCRBY 等命令。

    • d. 更新库存扣减记录表，添加新的扣减记录，包括订单ID、SKU、扣减的数量等信息。

    • e. 释放分布式锁。

乘客下单的流程：

  1. 在乘客下单之前，乘客选好起点和终点后，会预估驾驶路线和预估订单金额。
  2. 乘客点击下单后，服务端会生成订单信息，设置订单的状态为等待接单。
  3. 启动搜索附近司机的服务：
    1. 司机在点击接单按钮后，会上传自己的位置信息到redis中，之后会以轮询的方式一直上传司机位置信息到redis里，直到司机停止接单。
    2. 启动搜索服务，查找附近3公里内的司机。
    3. 根据司机的自定义设置，把符合要求的司机放到一个列表中。
  4. 使用定时任务，每分钟进行一次搜索附近司机，超过20分钟还没司机接单，就取消订单。
  5. 在搜索到的符合要求的司机列表中，给每个司机创建一个临时队列，过期时间为1分钟，司机开始接单时会在队列中找到订单数据，进行抢单操作。

司机接单的流程：

1、判断司机认证状态，与人脸识别，只要认证通过且这一天做过了人脸识别的司机才能接单

2、更新司机全局接单状态

3、删除司机之前的geo数据（重新做初始化，防止数据影响）和清空司机新订单临时队列（重新做初始化，防止数据影响）

4、实时轮询上传司机位置信息

5、轮询司机的临时队列，只要有合适的新订单产生，就能获取新订单数据。司机就可以进行抢单操作。

​ a. 判断redis中是否有订单

​ b. 司机抢单后修改订单的状态和设置抢到的司机的id

​ c. 删除redis中的订单的标识

在第5步中，需要加分布式锁来保证同一时间只有一司机能抢到订单。

乘客订单支付：

  1. 首先，当用户发起支付请求，返回支付页面。将支付请求的相关信息发送到 RocketMQ 的一个主题中。

  2. 消费者接收到支付请求消息后，首先解析消息内容，从中提取出订单ID、交易ID等相关信息。

    1. 检查订单状态：

      • 幂等性检查：根据交易ID查询数据库中是否存在已处理过的支付记录。如果交易ID已存在于数据库中，则直接返回已有的支付结果。如果不存在，新增支付记录。

      • 订单状态检查：如果交易ID不存在，再检查订单状态是否为未支付状态（例如NEW）。

    2. 如果订单状态为未支付，从订单服务中获取订单的基本信息。

    3. 如果使用了优惠券，需要从优惠券服务中获取优惠券详情，并检查该优惠券是否已被使用。如果优惠券未被使用，则更新优惠券的使用次数，并记录该优惠券对应的订单号等信息。

    4. 根据优惠券更新订单支付金额。

    5. 封装微信支付所需的参数，并调用微信支付接口发起支付。

  3. 这个支付处理逻辑包括了更新订单信息，更新优惠卷信息，新增支付信息等服务，所以要加一个分布式事务来保证数据的一致性。

  4. 支付请求处理完成后，将支付结果（成功或失败）发送到一个主题中。

    1. 创建另一个消费者服务来监听支付结果，并处理支付结果。

    2. 消费者接收到支付结果消息后，解析消息内容，从中提取出订单ID、支付状态等相关信息。

    3. 根据支付结果更新订单状态，例如将订单状态设置为“已支付”或“支付失败”。

如何实现附近司机搜索服务的？

首先这个搜索服务会搜索附近5公里内的司机，根据乘客的地理坐标位置和5公里半径，使用redis的geo数据类型，查询搜索范围内的司机的信息。然后根据这个司机的基本信息，判断这个司机是否符合这个订单的接单条件。对满足要求的司机进行封装返回。

乘客下单后的订单的处理：

  1. 用户下单时，前端界面可以通过异步请求（如 AJAX）提交订单，后端服务接收到请求后，立即返回订单的基本信息给用户界面。

  2. 后端服务将订单信息封装成消息并提交给生产者线程池中的线程处理，线程池中的线程负责将消息发送到 Kafka 集群中的一个主题。

  3. 在消费者端，也使用一个线程池来管理订单服务的实例。每个线程可以是一个独立的订单服务实例，负责订单的逻辑处理。

  4. 消费者拿到消息后，先在reids中插入一条记录表，设置记录的状态、过期时间，之后判断是否插入成功，插入成功，执行业务逻辑：

    • 计算驾驶路线。

    • 预估订单费用。

    • 更新订单信息到数据库。

    • 寻找附近的司机。把订单信息保存到司机的一个临时队列里。

  5. 插入失败，检查记录状态，记录状态为成功，返回消息消费成功；记录状态为正在进行，丢弃消息。

  6. 判断业务逻辑是否有异常，有异常，删除redis中的记录并进行重试。

  7. 没有异常，检测redis中的记录，如果过期了，重新获取消息并开始新的处理流程。没过期，更新redis中的记录状态。

  8. 最后插入记录到数据库。

  9. 使用seata来保证执行业务逻辑、更新redis中的记录状态和操作数据库这三个操作是原子执行的。

异常处理：

在消费者处理逻辑过程中，会进行异常捕获，根据异常信息，来进行重试机制，把消息发送到一个重试的主题，消费重试主题中的消息进行重试操作，利用偏移量和分区来追踪消息的重试次数。达到消息重试上限，就把消息发送到一个死信主题中。一个线程会把这个处理失败的消息保存到Mysql中的一个专门用于处理失败消息的表里。表中包括失败原因、重试次数，消息内容等信息。

优化：

  1. 在上述订单处理流程中，可以考虑在以下几个步骤引入缓存（如 Redis），以减轻数据库压力，提高查询效率：
  • 预估订单金额：如果预估订单金额的计算涉及到复杂的业务规则或依赖其他系统的数据，可以将计算结果缓存在 Redis 中。下次遇到相同的订单信息时，可以直接从缓存中获取预估金额，而无需重复计算。
  • 更新订单信息：在处理完各个消费者组的任务后，可以先将更新后的订单信息暂存到 Redis 缓存中，然后再批量同步到数据库中。这种方式可以减少数据库的写入频率，提高系统的整体性能。注意，这里应该给缓存设置一个过期时间，允许短时间的不一致问题。为了防止缓存穿透，可以使用布隆过滤器来避免。
  1. 在异常处理中，可能会有死信消息积压的问题，可以使用定时任务，定期去同步数据到Mysql中。
  2. 在处理失败的消息时，如果将其保存到数据库中的失败消息表，需要注意保持数据一致性。例如，如果在保存失败消息的同时还要更新订单状态，那么应该使用事务来确保这两个操作要么都成功，要么都失败。
